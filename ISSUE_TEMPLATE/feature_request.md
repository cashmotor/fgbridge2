**功能描述**
授权卡片状态自动失效：当系统发出需要用户授权的高危操作卡片后，如果用户未点击卡片按钮而是直接在会话中输入了新的文字信息，系统应自动将该授权卡片设为失效状态。

**动机/背景**
解决用户跳过授权直接进行新对话时，历史授权卡片依然有效的问题。这可以防止用户在之后的某个时间点误操作点击到之前的授权按钮，从而触发非预期的指令执行，提升系统的安全性。

**预期行为**
1. 系统发出授权卡片。
2. 用户发送新文本消息。
3. 系统在处理新消息前，通过飞书 API (如 Patch Message) 更新之前的授权卡片 UI，将其置灰或显示“已失效”。
4. 即使点击失效卡片的按钮，后端也应拒绝执行相关逻辑。

**可能的实现方案（可选）**
在 Router 接收到新消息时，查询该 Topic 是否存在 pending_confirm。如果存在，调用 Feishu API 更新卡片并从持久化存储中移除对应的确认 ID。

**替代方案（可选）**
无。

**附加信息**
相关需求：确保授权动作的原子性和时效性。

**功能描述 2：引入“内存热缓存 + 磁盘持久化”双级会话管理机制**

**动机/背景**
目前每轮对话都执行强制 `session/save` 和 `session/load`，在连续对话场景下会产生不必要的 IO 开销和响应延迟。通过引入二级缓存，可以显著提升系统的 TTFT（首字延迟）并优化资源消耗。

**预期行为**
1. **内存命中优先**：当 Router 发送请求时，若对应的 ACP 进程实例已在内存中载入该 Session，则跳过 `session/load` 环节。
2. **自动持久化**：
   - 在 `ACPDispatcher.cleanup` 执行进程回收前，自动触发 `session/save`。
   - 在 Python 服务正常退出时，触发全局 Session 持久化落地。

**可能的实现方案（可选）**
- 在 `ACPProvider` 中维护已加载 Session 状态。
- 修改 `Router._route_to_role` 增加判断。
- 增强 `ACPDispatcher.cleanup` 逻辑。
